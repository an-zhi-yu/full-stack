# 业务模块从接口到代码：几大块与推荐顺序（前端对照）

面向：习惯写「路由 + service + store」的前端同学，在 Spring Boot 里落地一个**资源模块**（例如用户、文章）时，脑子里可以按下面几块拆。

---

## 1. 先定「对外契约」

- **URL + HTTP 方法**：列表 / 详情 / 新建 / 更新 / 删除 是否齐全，路径里 id 怎么放（`/users/{id}`）。
- **请求体 / 响应体 JSON 形状**：哪些字段必填、敏感字段（如 password）是否出现在响应里。
- **状态码约定**：团队是「REST 风格」还是「HTTP 永远 200、看 body.code」——前后端要一致。

对应代码里通常是：`Controller` 的方法签名 + `DTO` / `record` 类型。

---

## 2. 分层：谁负责什么

| 层次 | 职责（一句话） | 前端类比 |
|------|----------------|----------|
| **Controller** | 解析 HTTP（路径、Query、Body、Header），调 Service，决定 **HTTP 状态码** | Express `router`，尽量薄 |
| **Service** | 业务规则：校验、编排、生成 id、事务边界（以后接 DB） | 业务 service / hook 里的「纯逻辑」 |
| **Repository** | 数据怎么存取（本案例是内存 Map + JSON；生产多为 JPA/JDBC） | 封装 `fetch` / ORM 的那一层 |
| **Entity** | 领域模型，**可含密码**等内部字段 | 服务端「真实一行数据」 |
| **DTO** | 对外传输形状，**登录响应、列表项、更新请求**可各用一类，避免把 Entity 直接当 API 返回 | TypeScript `type` / Zod schema |

---

## 3. 推荐落地顺序（小步可跑）

1. **DTO +（可选）Entity**：先把 JSON 形状定下来。  
2. **Repository**：假数据或内存 Map，能 `save/findById`。  
3. **Service**：组合 Repository，返回 DTO（敏感字段不暴露）。  
4. **Controller**：接上路径，返回 `ApiResult` 或 `ResponseEntity`。  
5. **测试**：MockMvc 打一两个 happy path。  
6. **鉴权**：需要登录的接口再交给 Filter / Security（见 `JWT鉴权与Filter规则.md`）。

---

## 4. 与「接口文档」的关系

- 仓库里的 **`api-examples.http`**、**`REST-博客文章API.md`** 就是契约的落地版；改 URL 或字段时，记得同步这两处，避免前端联调踩坑。

---

## 5. 本仓库已对齐的实践

- 统一响应体 **`ApiResult<T>`**（`code` / `message` / `data`）。  
- 用户模块：**Entity 含密码，DTO 不含**；MapStruct **`UserMapper`** 做 Entity→DTO。  
- 文章模块：**浏览 / 点赞** 与正文数据分离在 **`PostEngagementStore`**，避免改 JSON 源文件结构。
